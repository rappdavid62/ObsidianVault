# Shared Project Context Notes

This section distills the project’s recurring direction from the shared project conversations.

## Repeated project goals

- Build a clean depression project that can be shared with other people
- Remove embarrassing or distracting material from the shareable package
- Build a full five-year depression framework
- Separate preamble / preparation, implementation, years 1-5, maintenance, and worksheets
- Make the project usable for low-energy people rather than intellectually impressive but unusable

## Repeated content themes

### 1. Depression is more than sadness
The project repeatedly emphasized grief, sadness, anhedonia, low energy, flattened experience, functional collapse, and distorted self-perception.

### 2. Function matters as much as mood
An important recurring distinction in the conversations was:
a person may describe mood vaguely, but daily functioning may still be collapsing.
The project therefore tracks work, sleep, hygiene, relationships, energy, finances, and follow-through.

### 3. Distinguish depression from a merely unsatisfactory life
The conversations repeatedly pushed the need to distinguish:
- a bad or unsatisfying life situation
from
- a chronic depressive process that distorts interpretation, motivation, energy, and self-modeling

### 4. Keep the plan low-friction
The user repeatedly pushed for a plan that survives:
- low energy
- work obligations
- limited money
- schedule instability
- collapse days
- social strain
- plain life mess

### 5. Use repetition and anchoring
The project context repeatedly highlighted:
- mantra repetition
- meditation
- circadian stabilization
- perspective shifts
- repeated small anchors rather than one-time insight

### 6. Anti-self-confabulation / self-distortion theme
The project conversations also explored negative self-modeling and the way depression distorts reality, identity, and interpretation.
That theme appears to sit behind the project’s insistence on:
- concrete review
- externalized worksheets
- behavioral tracking
- function-first interpretation
- procedure instead of self-attack

### 7. Need for personalization
The project was not meant to be one generic lecture.
The questionnaire and front-end modules were intended to tailor the opening message to:
- symptom burden
- constraints
- tone tolerance
- what the intro needs to do first
- realistic available capacity

## Shareability rule

One of the clearest practical project constraints across multiple conversations was:
make the package shareable without dragging along embarrassing searches, messy side trails, or irrelevant content.

That rule guided the structure of this export.
